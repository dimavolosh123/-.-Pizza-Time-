# Экзаменационный проект по предмету ООП: Pizza-Time

Разработано консольное приложение на Python, реализующее бизнес-логику работы пиццерии. Этот проект был создан в рамках экзаменационного задания по курсу «Объектно-ориентированное программирование» в Институте прикладных информационных технологий СГТУ имени Гагарина Ю.А.

## Описание данного проекта

Разработанное приложение воспроизводит полный процесс обработки заказа в предприятии общественного питания — от возникновения у клиента желания заказать пиццу до её приготовления и вывода уведомления на электронное табло. Архитектура решения базируется на фундаментальных принципах ООП, обеспечивающих чёткое распределение обязанностей между автономными модулями системы.

## Архитектура и сущность

В состав программы входят следующие основные классы:

1. `User` — клиент, имеющий имя. Управляет инициацией заказа, получает уведомления с табло и забирает готовую пиццу.
2. `Pizzeria` — центральный элемент системы, отвечающий за приём заказов, их регистрацию в общем пуле, координацию приготовления и отправку сигналов на табло.
3. `Pizza` — сущность, хранящая информацию о блюде (например, название).
4. `Order` — связующий объект, содержащий уникальный идентификатор, ссылки на клиента (User) и заказанную пиццу (Pizza), а также актуальный статус готовности.
5. `UI` — вспомогательный класс со статическими методами для эстетичного форматирования консольного вывода, реализующий принцип единственной ответственности (SRP).

## Принципы ООП иприминение их в Python

В процессе разработки были реализованы и показаны на практике следующие принципы:

* **Инкапсуляция** и сокрытие данных. Все ключевые внутренние атрибуты классов (например, пул заказов `__orders_pool` или имя пользователя `__name`) защищены с помощью механизма *Name Mangling* (двойное подчёркивание). Доступ к ним для чтения организован через декораторы `@property`, что обеспечивает контролируемое взаимодействие.
* **Полиморфизм и слабая связанность**. Пиццерия оперирует с заказом как с абстрактным объектом, вызывая методы уведомления у привязанного к нему клиента, при этом не имея прямой зависимости от внутренней реализации класса пользователя.
* **Принцип DRY (Don't Repeat Yourself).** Код для вывода разделителей и заголовков интерфейса вынесен в отдельный класс `UI`, что предотвращает многократное повторение вызовов `print()` в основном теле программы.
